# Баг 02 — Удалённый доступ через Tailscale Funnel + краснеющий пульт + окно консоли при старте

**Статус:** 🟡 фикс применён (2026-07-04), ждёт подтверждения владельца с телефона.
**Компоненты:** Tailscale Funnel · Caddy (reverse-proxy) · Homepage (пульт) · llama-swap (автозапуск).

---

## Симптомы (со слов владельца)

1. **Funnel с телефона — пароль, потом пустота.** Ссылка `https://krinikspc.forest-ratio.ts.net/`
   спрашивает пароль (это basicauth Caddy), а затем «ничего не может открыть». Подозрение владельца —
   в том, как реверс-прокси обслуживает контейнеры. («Если Caddy не подходит — меняем».)
2. **Пульт краснеет со временем.** Даже на `http://localhost:3005/` через некоторое время у виджетов
   появляется «API Error», зелёные миллисекунды пинга сменяются красным `ERROR`.
3. **Консоль при старте ПК.** При запуске компьютера открывается окно консоли (лог llama-swap).
   Не нужно — KLAS должен стартовать тихо в фоне.

Пример лога из окна (симптом 3): llama-swap слушает `127.0.0.1:8080`, отдаёт `/ui/`, `/health`,
`/v1/models` — сервер здоров, проблема только в видимом окне.

---

## Диагноз (2026-07-04)

**Симптом 1 — нет маршрута для корня `/`.** В `caddy/Caddyfile` описаны только `handle_path /llm/*`,
`handle /wiki*`, `handle /homepage*`. Корень `/` попадает в общий `handle {}` (catch-all с basicauth):
пароль спрашивается, но ни один `reverse_proxy` внутри не матчится → Caddy отдаёт пустой ответ.
Реверс-прокси исправен — просто корню некуда вести. (Подтверждено: `curl :443/` → 401 на basicauth,
после авторизации — пусто.)

**Симптом 3 — окно консоли.** Задача Планировщика `KLAS llama-swap` запускает
`cmd /c F:\KLAS\llama-swap\start-llama-swap.bat` при входе в систему. `cmd` при onlogon-триггере
показывает окно консоли, в котором виден лог llama-swap. `llama-swap.exe` крутится в интерактивной
сессии Console — отсюда видимое окно.

**Симптом 2 — краснота от падения llama-swap (связан с симптомом 3).** В логах контейнера homepage —
`ECONNREFUSED` на `192.168.65.254:8080` (= `host.docker.internal:8080` = llama-swap на хосте).
siteMonitor пульта пингует `http://host.docker.internal:8080/health`; когда процесс llama-swap не
отвечает (окно закрыли, процесс упал, ещё не стартовал) — монитор краснеет. Проверено: все три
siteMonitor-эндпоинта (open-webui, kiwix, llama-swap) сейчас отдают `200`; `/health` llama-swap отдаёт
`200` даже без загруженной модели → ttl-автовыгрузка мониторы НЕ роняет. Значит краснота — от
нестабильности самого процесса llama-swap, живущего в хрупком окне консоли.

→ **Симптомы 2 и 3 — одна корневая причина:** llama-swap запускается видимым и хрупким. Тихий,
фоновый, самовосстанавливающийся запуск чинит оба.

---

## Фикс (2026-07-04)

1. **Корневой редирект (симптом 1).** В `caddy/Caddyfile`, внутри `handle {}` после basicauth добавлен
   `redir / /homepage/` — открытие funnel-URL после пароля ведёт на пульт. Caddy перезагружен
   (`docker exec caddy caddy reload`).
2. **Тихий фоновый запуск llama-swap (симптомы 2+3).** Добавлен скрытый лаунчер
   `llama-swap/start-hidden.vbs` (`WScript.Shell.Run ..., 0` — окно скрыто). `tools/install-autostart.ps1`
   переписан: задача Планировщика вызывает `wscript //B start-hidden.vbs` (без окна) и настроена на
   авто-рестарт при сбое (3 попытки, интервал 1 мин) — llama-swap самовосстанавливается, монитор пульта
   остаётся зелёным. Задача пересоздана, старый видимый процесс остановлен и перезапущен тихо.

## Известное ограничение / на будущее

- **Ссылки-виджеты пульта ведут на `http://127.0.0.1:…`** — с телефона (через funnel) они не откроются
  (127.0.0.1 = сам телефон). Open WebUI (:3080) и llama-swap UI (:8080/ui) вообще не проксируются через
  Caddy/funnel — удалённо к ним не подключиться. Это отдельный вопрос дизайна удалённого доступа (что
  выставлять наружу и под какой авторизацией), а не этот баг. Кандидат на идею/интервью.
- Ярлык `tailscale_funnel_443.bat` в папке автозагрузки мелькает окном при старте. Funnel уже
  персистентен через `tailscale funnel --bg` (восстанавливается tailscaled сам) — ярлык избыточен;
  можно убрать (по решению владельца), чтобы совсем не мелькало.

## Как проверить

- **Симптом 1:** с телефона открыть `https://krinikspc.forest-ratio.ts.net/`, ввести логин/пароль —
  должен открыться пульт (Homepage).
- **Симптом 2:** оставить пульт открытым; мониторы llama-swap должны держаться зелёными (в т.ч. после
  простоя >5 мин и после автовыгрузки модели).
- **Симптом 3:** перезагрузить ПК / выйти-войти — окно консоли не должно появляться; `curl
  http://127.0.0.1:8080/health` → `200`.
