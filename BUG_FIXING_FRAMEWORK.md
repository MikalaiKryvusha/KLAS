# BUG_FIXING_FRAMEWORK — как агент чинит дефекты

Чтобы починить баг, агент обязан:

- **Фокусироваться только на этом одном баге.** Не рефакторить мир; не чинить три других вещи по пути.
- **Рассказывать в чате**, хотя бы немного, естественным языком — что делаешь прямо сейчас, — чтобы
  человек мог мельком глянуть и понять, где идёт работа.
- **Рефлексировать и фиксировать знание** по каждому багу, даже мелкому, в отдельном markdown-файле
  на баг в директории `bugs/`.
- **Войти в цикл:** запустить систему → воспроизвести баг → прочитать логи по этому багу → сформировать
  гипотезу причины → внести *одно точечное* изменение → перезапустить → снова попытаться воспроизвести.
- Суть: **точечные изменения, затем проверка — помогло изменение или нет.**
- Упрощённо: Фикс → Тест → Чтение логов → Фикс → Тест → Чтение логов → … пока не заработает правильно.
  Правильная работа **и есть критерий приёмки** — в этот момент баг считается починенным.
- Для починки бага часто полезно **искать решение в вебе** — форумы, GitHub issues, Reddit,
  Stack Overflow, официальная документация.

> ⚠️ **ПРАВИЛО 3 ПОПЫТОК → переход к исследованию (`/bug-research`).**
> Если после **трёх итераций** «точечный фикс → запуск → тест» баг НЕ починен — СТОП, хватит тыкать
> вслепую. Дальнейшие случайные попытки тратят время и запуски, а (если тест зависит от внешних
> сервисов) ещё и ненадёжны. Вместо этого запусти скилл **`/bug-research`**: глубокий веб-поиск с
> записью сырой базы знаний в документ бага, чтение и анализ кода/конфигов БЕЗ правок для локализации
> причины, рефлексия и обоснованная гипотеза + план. Возвращайся к фиксам только когда понял причину.
> Скилл: `.claude/skills/bug-research/SKILL.md`.
>
> 🧠 **И, САМОЕ ГЛАВНОЕ (см. `PHILOSOPHY.md`):** долгое застревание почти всегда означает, что ты
> делаешь ФИКС СЛИШКОМ СЛОЖНО, потому что НЕ ПОНЯЛ задачу, — а НЕ что задача сложна или библиотека
> «глючная». Библиотеки просты; всё придумано до нас. Если фикс громоздкий, с кучей флагов и обходов —
> это красный флаг: остановись и ПЕРЕПОЙМИ задачу простыми словами, найди простой поддерживаемый путь
> (KISS + Оккам). Застревание = сигнал упростить понимание, а не «копать сильнее».

- Для эффективной починки код и конфиги должны быть **прокомментированы**.
- Для эффективной починки ОБЯЗАТЕЛЬНО рефлексируй и записывай знание о работе над багом в отдельный
  документ именно про *эту* работу. Такие размышления живут отдельными markdown-документами в `bugs/`.
- Чтобы взаимодействовать с проверяемой системой, можно и нужно **ставить или строить дополнительный
  инструментарий**, который позволяет агенту наблюдать и управлять софтом: видеть вывод, инспектировать
  состояние, детерминированно воспроизводить дефект и прогонять систему без человека. Инвестируй в эту
  инструментовку — она окупается на каждом следующем баге.

## Инструментовка — строй тестовый харнесс, не гадай

Самый большой множитель силы для автономной отладки — **харнесс**: инструментарий, позволяющий агенту
воспроизводить и наблюдать баг самостоятельно, без человека в цикле и без ненадёжной ручной проверки.

Принципы:

- **Никогда не гадай, что делает программа, — наблюдай её.** Добавь структурированное логирование,
  отладочный командный канал, детерминированный способ привести софт ровно в то состояние, которое
  триггерит баг.
- **Делай воспроизведение детерминированным.** Баг, который воспроизводится по требованию, — это баг,
  который можно починить. Баг, который «иногда бывает», — сначала задача на исследование
  (`/bug-research`).
- **Предпочитай объективную проверку взгляду на глазок.** Если визуальная/ручная проверка ненадёжна
  (тонкие искажения, тайминги) — придумай объективную (контрольный вход известной формы, измеримый
  выход, лог размеров/чексумм) и запиши её в документ бага.
- **Наращивай харнесс со временем.** Каждый раз, когда делаешь что-то руками для воспроизведения или
  проверки, спроси: «могу ли я добавить команду/флаг/скрипт, чтобы в следующий раз это был один шаг?»
  Добавь — и задокументируй в `AGENT_GUIDE.md`. Харнесс — живой инструмент: расширяй и документируй.

## Работа с логами

- Перед чтением всегда сверяй **текущее время** и **таймстемп лог-файла/записей** — читай свежие,
  релевантные логи, а не вчерашние.
- Фильтруй логи под баг: grep по нужной подсистеме, тексту ошибки, маркерам падения.
- Прикладывай ключевые строки (stack trace, сообщение об аварии, коды ошибок, размеры) в документ
  бага — это криминалистика, на которую будут опираться будущие сессии.

## Документ бага (один на баг, в `bugs/`)

Фиксируй, а не пересказывай. Хороший документ бага позволяет будущей сессии (с пустым контекстом)
подхватить баг «с холода». Каноническая структура (полный шаблон — в скилле `/report-bug`):

```
# Bug NN — <описание одной строкой>

**Статус:** 🔴 OPEN  (или 🟡 частично / 🔬 только исследование / 🔧 фикс ждёт проверки / ✅ DONE)
**Версия/сборка:** <...>   ·   **Когда/контекст:** <дата, в ходе какой задачи найден>

## Симптом
<что наблюдается и чем отличается от ожидаемого>

## Воспроизведение (детерминированное)
<шаги; команды харнесса, если есть>

## Криминалистика
<ключевые логи / падение / измерения>

## Корневая причина / Гипотезы
<причина, если известна; иначе ранжированные гипотезы — НЕ патчить вслепую>

## План фикса (или сам фикс, если сделан)
<шаги; связь с архитектурой / другими багами>

## Ссылки
<связанные баги / идеи / интервью>
```

Когда баг подтверждённо починен и проверен — помечай его DONE по конвенции DONE-тега (переименуй
`bugs/NN_x.md` → `bugs/NN_DONE_x.md` и добавь секцию `## ✅ STATUS: DONE (дата)`). См.
`AGENT_GUIDE.md` → «Беклог и DONE-тег» и скилл `/check-backlog`.

## Если баг в чужой библиотеке

Если нашёл настоящий дефект в сторонней зависимости — заведи issue/тикет в её трекере (например, через
`gh` CLI, от имени человека, если авторизовано) и сошлись на этот тикет из своего документа бага. Это и
помогает экосистеме, и документирует, почему ты сделал обход.
