# Идея 11 — знания Kiwix в чатах Open WebUI

**Статус:** ✅ реализовано (2026-07-04), тул готов и проверен — ждёт разовой установки владельцем в
UI Open WebUI (тулы Open WebUI живут в его БД, а не в файлах, поэтому добавляются через веб-интерфейс).
Одобрено владельцем в чате.

## Задача

LLM в Open WebUI должен уметь пользоваться нашей локальной базой Kiwix (Википедия и другие `.zim`).
Требование владельца: искать **по всем книгам всех языков**, и чтобы **новый `.zim`, докинутый в
Kiwix, автоматически становился видимым** без правки инструмента.

## Почему не «база знаний» Open WebUI

«Базы знаний» Open WebUI — его внутренний RAG (загруженные документы). Kiwix — отдельный сервер
`.zim`. Автоматически одно в другое не попадает (см. баг 03). Заливать всю Википедию в RAG
непрактично. Правильный путь — дать модели **инструмент** (function calling), который ищет в Kiwix
на лету. Ровно это модель и пыталась делать (в баге она звала `list_knowledge_bases`).

## Решение — Tool `search_local_knowledge`

Файл: `open-webui/tools/kiwix_search.py` (версионируется). Логика:

1. На каждый запрос берёт список языков из каталога Kiwix (`/catalog/v2/entries` → `<language>`),
   поэтому любой новый `.zim` (его язык уже в каталоге) попадает в поиск **автоматически**.
2. Ищет по каждому языку (`/search?pattern=…&books.filter.lang=<lang>`) — так покрываются ВСЕ книги
   (Kiwix не умеет «все книги» одним запросом без указания книг, но фильтр по языку это решает).
3. Тянет текст верхних статей (`/content/<книга>/<статья>`), снимает HTML, отдаёт модели с указанием
   источника. UTF-8 форсируется явно (Kiwix отдаёт без charset).

Настройки (Valves в туле): базовый URL Kiwix (`http://kiwix_wikipedia:8080/wiki` — внутри docker-сети),
число результатов на язык, число статей и длина текста. Зависимость — `requests` (есть в Open WebUI).

**Проверено (2026-07-04):** из контейнера open-webui запрос «Эйнштейн» → найдены статьи в
`wikipedia_ru_all` и `wikisource_ru_all` (кросс-книжно), текст корректный (UTF-8).

## Установка (делает владелец, один раз)

**Способ А — импорт JSON (проще, рекомендую).** Готовый файл: `open-webui/tools/kiwix_search.tool.json`.
1. Open WebUI → **Workspace → Tools** → кнопка **Import Tools** (↑) → выбрать `kiwix_search.tool.json`.
2. Всё — тул появится в списке (`specs` Open WebUI построит из кода сам).

Формат JSON — это массив тулов ровно как при **Export Tools**; импорт читает поля `id`, `name`, `meta`,
`content`, `access_grants` (проверено по коду Open WebUI 0.10.2). Файл генерируется из `.py`:
`py -3 tools/build-owui-tool-json.py` (или см. коммит идеи 11).

**Способ Б — вставить код руками.** Workspace → Tools → **+**, вставить содержимое
`open-webui/tools/kiwix_search.py`, Сохранить.

**После установки (любой способ):** в чате включить инструмент (значок ⚙️/«+» у поля ввода → «Kiwix —
локальная база знаний») либо привязать его к модели в её настройках. Спросить что-нибудь
энциклопедическое — модель вызовет `search_local_knowledge` и ответит по локальной Википедии.
(Нужна модель с tool calling — наши Qwen проходят.)
