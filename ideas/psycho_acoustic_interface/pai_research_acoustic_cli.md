## 0. TL;DR для реализации

1. **Режим B («акустический CLI») — зрелая инженерия, но требует ОБУЧЕНИЯ пользователя.** Это не «команды
   в мозг», а хорошо изученный **не-речевой аудио-UX**: earcons, auditory icons, spatial audio,
   сонификация. Применяется в авиации (NASA), помощи незрячим (vOICe), HCI. Ценность для KLAS —
   быстрые фоновые **уведомления** Jarvis без слов (готово/ошибка/прогресс/статус).
2. **Стек генерации звука — двухуровневый:**
   - **Простое и достаточное (рекомендую старт):** **Web Audio API** (в вебе/Electron) или его
     Node-эквивалент — прямые `OscillatorNode` + `GainNode`. Для entrainment этого хватает с запасом.
     Библиотеки: **Tone.js** (высокоуровневый синтез), **BinauralBeatJS**.
   - **Оффлайн-рендер в файл (CLI, детерминированно):** **SBaGenX** (форк SBaGen+) — бинауральные/
     монофонические/изохронные, скриптуемые сессии, качественный оффлайн-рендер. Идеален как «движок
     протоколов» под управлением Jarvis. Плюс **ffmpeg** для микса/кодирования.
   - **Продвинутая сонификация (если понадобится режим B по-взрослому):** **SuperCollider** (scsynth +
     OSC) или **Pure Data** — но это уже отдельный тяжёлый стек, тянуть только при явной необходимости
     (принцип простоты KLAS).
3. **Приоритет стека — Node/JS** (как весь KLAS). Python/нативное — только если задача решается им
   заведомо лучше (SuperCollider, тяжёлый DSP).

## 1. Режим B: «акустический CLI» — что это на самом деле

Речь ~30–60 бит/с — медленный интерфейс. Не-речевой звук передаёт короткие сообщения быстрее и в
обход речевых центров. Пять приёмов (из консультации + гуглёж):

### 1.1. Earcons (звуковые токены)
- Короткие (50–150 мс) **абстрактные синтезированные** звуки с жёсткой, заученной семантикой.
- Каноничная база — диссертация **Stephen Brewster** (Glasgow): музыкальный синтаксис команд через
  тембр, ритм, высоту, огибающую (envelope); **параллельные earcons** (несколько «команд» одновременно).
  Кодируют древовидные структуры (иерархию меню/статусов).
- Требуют **обучения** (пользователь учит «язык»), но потом считываются на уровне рефлекса.

### 1.2. Auditory Icons (звуковые иконки)
- В отличие от earcons — **метафорические, узнаваемые из жизни** звуки (звук «в корзину» = сминание
  бумаги). Не требуют обучения, но ограничены тем, для чего есть естественная метафора.

### 1.3. Spatial Audio (пространственное кодирование)
- Команда/источник привязаны к 3D-координате вокруг головы (азимут/высота). Мозг локализует источник
  **быстрее, чем декодирует смысл** → «приоритет» и «канал» кодируются позицией.
- Снижает **Dual-Task Cost** (издержки переключения задач) — доказано в HCI (Oxford).
- Реализация: HRTF-панорамирование (в Web Audio — `PannerNode`).

### 1.4. Частотное разделение каналов (Auditory Stream Segregation)
- Несколько ритмических паттернов на разнесённых частотах (200/1000/5000 Гц) слух не смешивает →
  параллельные независимые «каналы» статуса.

### 1.5. Сонификация (Sonification)
- Перевод **данных** в аудиопараметры (аналог графика в терминале): значение переменной → частота среза
  шума, высота, темп, «текстура». Пользователь слышит статус фоном, не глядя на экран.
- Метод **TaDa** (Task and Data analysis, Stephen Barrass) — как сопоставлять типы данных с
  психоакустическими параметрами.
- Классика применения — **cockpit sonification** (NASA Ames): телеметрия как аккорд (штатно = консонанс,
  сбой = диссонанс, внимание за миллисекунды).

### 1.6. Sensory Substitution (для контекста)
- **vOICe** (Peter Meijer): картинка с камеры → звук (высота = Y, время = X, громкость = яркость).
  Показывает потолок канала: при обучении слуховая кора начинает «показывать» картинку. Для KLAS —
  далёкий горизонт, не сейчас.

> **Практический вывод для KLAS:** режим B в первой версии — это **набор из ~10–20 earcons + пара
> сонификаций** для событий Jarvis/KLAS: «задача выполнена», «ошибка», «думаю…», «долгий процесс идёт»
> (сонификация прогресса), «новое уведомление», «модель загрузилась». Полноценный «CLI-язык» и
> spatial-мультиканальность — задел на будущее.

## 2. Инструменты генерации звука (гуглёж)

### 2.1. Готовые генераторы entrainment (референс + возможный движок)
| Инструмент | Что | Лицензия | Годен для KLAS как |
|-----------|-----|----------|--------------------|
| **SBaGenX** ([github lm7137](https://github.com/lm7137/SBaGenX)) | CLI-генератор: бинауральные, монофонические, изохронные, микс; скриптуемые сессии; оффлайн-рендер | форк SBaGen+ (GPL-родословная) | **движок протоколов** под Jarvis: описал протокол текстом → отрендерил файл |
| **SBaGen / SBaGen+** ([uazu.net](https://uazu.net/sbagen/)) | Оригинал: реалтайм-биения по 24-часовому скрипту-«sequence» | GPL, freeware | референс формата протоколов/секвенсов |
| **Gnaural** ([sourceforge](https://gnaural.sourceforge.net/)) | GUI/движок бинауральных + изохронных, программируемый | GPL | референс UX и модели «слоёв» |

> ⚠️ Лицензии GPL — если тянуть код в KLAS, помнить про совместимость (KLAS — MIT). Безопаснее
> использовать как **внешний бинарь-движок** (запускать процессом), а не встраивать исходники, ИЛИ
> написать свой генератор на Web Audio (тривиально для entrainment).

### 2.2. JS/Node-стек (рекомендуемый для «своего» генератора)
- **Web Audio API** — `OscillatorNode` (sine/square/triangle/sawtooth), `GainNode` (АМ/изохронные через
  модуляцию gain), `PannerNode` (spatial), стерео-разведение каналов (бинауральные). Для entrainment
  этого достаточно **полностью**.
- **[Tone.js](https://tonejs.github.io/)** — высокоуровневый синтез поверх Web Audio (осцилляторы,
  огибающие ADSR, сигналы-рампы частоты). Удобно для приятных протоколов и earcons.
- **[BinauralBeatJS](https://github.com/ichabodcole/BinauralBeatJS)** — готовая мини-библиотека
  бинауральных биений на Web Audio (референс/старт).
- **ffmpeg** — микс речи Jarvis + звуковой ландшафт + entrainment-слой; кодирование в файл для
  воспроизведения на Android.
- Рендер вне браузера: `node-web-audio-api` (нативная реализация Web Audio для Node) либо
  offline-рендер через headless-контекст.

### 2.3. Продвинутая сонификация (только если режим B «по-взрослому»)
- **SuperCollider** (scsynth-сервер + управление по **OSC**) — промышленный стандарт исследовательской
  сонификации; KLAS шлёт данные по OSC → сервер синтезирует. Тяжёлый отдельный стек.
- **Pure Data / Max/MSP** — визуальное программирование интерактивной сонификации.
- Тянуть **только** при явной необходимости — против принципа минимума сущностей KLAS.

## 3. Как это ложится в архитектуру KLAS/Jarvis

```
Голосовой интент пользователя
      │  «запусти протокол засыпания» / (событие KLAS: «сборка готова»)
      ▼
Jarvis (интент-роутер, навык /pai)
      │
      ├─ Режим A (состояние): выбрать пресет из библиотеки протоколов
      │        │  или сгенерировать по параметрам (θ 6 Гц, 20 мин, АМ+розовый шум, fade)
      │        ▼
      │   Генератор аудио (Web Audio/Tone.js  ИЛИ  SBaGenX как внешний движок)
      │        │  → поток / файл (+ ffmpeg микс с речью и ландшафтом)
      │        ▼
      │   Слой безопасности (лимит громкости, дисклеймер, контекст-чек)
      │        ▼
      │   Воспроизведение в наушники пользователя (Android-устройство)
      │
      └─ Режим B (уведомление): выбрать earcon/сонификацию из набора событий → короткий звук
```

- **Библиотека протоколов** = часть памяти Jarvis (CRUD навыков/протоколов из [idea 05](../05_Jarvis.md)):
  удачный сгенерированный протокол сохраняется, каталогизируется, переиспользуется.
- **Формат протокола** — декларативный (YAML/JSON): модальность, целевая частота(ы), несущие тоны,
  длительность/фазы, огибающие, фон (шум/эмбиент), лимит громкости, метаданные безопасности. Вдохновение —
  формат «sequence» SBaGen, но свой, читаемый и комментируемый (стиль KLAS).

## 4. Открытые инженерные вопросы
1. **Где играет звук?** Целевой сценарий — Android-устройство Криника с наушниками. Значит нужен канал
   доставки: (а) Jarvis рендерит файл на хосте → отдаёт на телефон; (б) генерация прямо на телефоне по
   параметрам от хоста. → зависит от будущей архитектуры Jarvis-клиента на Android.
2. **Свой генератор vs внешний движок (SBaGenX):** для режима A свой на Web Audio почти наверняка проще и
   без лицензионных вопросов. Решить на этапе прототипа.
3. **Микс с речью:** Jarvis может «говорить сквозь» entrainment-ландшафт (голос поверх фона) — приятный UX,
   но нужно следить за разборчивостью (психоакустика маскировки, гл. 3 Sonification Handbook).
