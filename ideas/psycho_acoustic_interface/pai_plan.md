## Принципы (из PHILOSOPHY.md и заметок владельца)

- **ПРОСТОТА (KISS + Оккам):** сначала минимальный работающий генератор состояний на Web Audio, а не
  тяжёлый DSP-стек. Сложность добавлять только когда простое доказанно не хватает.
- **Честность:** PAI — ассистивный аудио-UX, не «программирование мозга», не медицина. Тон и дисклеймеры
  соответствуют (см. research #1, раздел 2 и 4).
- **Безопасность — не опция, а слой Фазы 1.** Лимит громкости, дисклеймер, стоп-условия — с самого начала.
- **Минимум сущностей:** предпочесть свой генератор на Web Audio внешним GPL-движкам; тяжёлый стек
  (SuperCollider) — только под явную необходимость.
- **Заимствовать лучшее:** SBaGenX/Gnaural/Tone.js/BinauralBeatJS — как референс и, при нужде, внешний движок.

## Развилки для `/interview` ПЕРЕД реализацией (владельческие решения)

1. **Скоуп v1:** только режим A (состояния) — рекомендую — или сразу A + B (earcon-уведомления)?
2. **Генератор:** свой на Web Audio (рекомендую) или внешний движок SBaGenX процессом?
3. **Где рендер/воспроизведение:** на хосте с отдачей файла на Android, или генерация на самом Android?
   (зависит от архитектуры Jarvis-клиента — вероятно решится в плане Jarvis, не тут.)
4. **Дефолтная модальность приятных протоколов:** АМ поверх эмбиента (рекомендую) vs чистые бинауральные.

## Фаза 1 — Прототип генератора состояний (песочница на хосте) 🔲

**Цель:** на хосте (без Jarvis) собрать генератор, который по декларативному описанию протокола рендерит
аудио и играет его. Это доказывает науку и формат, ничего не завязывая на Jarvis.

- [ ] **Формат протокола** (декларативный YAML/JSON, комментируемый — стиль KLAS): модальность
      (binaural/monaural/isochronic/am), несущие частоты, целевая частота(ы) и фазы во времени,
      длительность, огибающие fade-in/out, фон (розовый/белый шум/эмбиент), потолок громкости,
      метаданные безопасности (`min_hz`, флаг «мягкое нарастание»).
- [ ] **Генератор** на **Web Audio** (Tone.js/BinauralBeatJS как основа): все 4 модальности (research #2, §3);
      дефолт приятных протоколов — АМ поверх мягкого шума/эмбиента.
- [ ] **Оффлайн-рендер в файл** (`node-web-audio-api` offline-контекст → WAV/opus через ffmpeg) +
      потоковое воспроизведение для быстрой отладки.
- [ ] **Слой безопасности v1** (research #1, §4 чек-лист): жёсткий лимит громкости (именованная константа),
      обязательные fade, пометка протоколов <10 Гц, авто-таймаут, мгновенный стоп.
- [ ] **Тестовый харнесс:** скрипт `tools/pai-render.mjs <protocol.yaml> [out]` — рендерит и, опционально,
      анализирует спектр (проверить, что целевая частота реально присутствует). Строку — в таблицу
      харнесса `AGENT_GUIDE.md`.
- [ ] **Дисклеймер** (текст) + запись в документ: PAI не медицинский прибор.

**Проверка фазы:** отрендерить «протокол засыпания» (θ 6 Гц → δ, 20–30 мин, АМ+розовый шум, fade),
спектральным анализом подтвердить целевую модуляцию, прослушать, убедиться в комфортной громкости.

## Фаза 2 — Библиотека протоколов (пресеты) 🔲

**Цель:** каталог готовых, откомментированных протоколов «из коробки».

- [ ] Пресеты: **засыпание** (θ→δ, длинный), **расслабление/сброс** (α, 10–15 мин), **фокус**
      (β/АМ поверх эмбиента, рабочая сессия), **бодрость** (изохронные, короткий). Каждый параметр — с
      комментарием «почему такое значение» (частоты из research #1, §1).
- [ ] Метаданные каждого пресета: назначение, длительность, модальность, требования (наушники?),
      безопасность, контекст-ограничения (напр. «не за рулём»).
- [ ] Каталог как часть будущей **библиотеки памяти Jarvis** (CRUD): структура папок/индекс, чтобы Jarvis
      находил протокол по смыслу запроса.

**Проверка:** 4+ рабочих пресета, каждый рендерится и звучит осмысленно; владелец слушает и даёт фидбэк.

## Фаза 3 — Интеграция в Jarvis (навык `/pai`) 🔲

> Возможна только когда базовый Jarvis существует (голос in/out + навыки + звук в наушники).

- [ ] **Навык `/pai`** в системе навыков Jarvis: интент «запусти протокол X» → выбор пресета из библиотеки
      или генерация по параметрам → воспроизведение в наушники пользователя.
- [ ] **NLU-маппинг:** «хочу быстро уснуть» → протокол засыпания; «нужно сосредоточиться» → фокус; и т.д.
      (Jarvis понимает намерение, не требует точного имени пресета).
- [ ] **Голосовое управление сессией:** «Jarvis, стоп» (мгновенный гейт), «тише/громче», «продли».
- [ ] **Контекст-чек безопасности** для сна/глубокого релакса: голосовое подтверждение «ты в безопасном
      месте?» (сценарий «Криник гуляет по городу в наушниках» — режим сна опасен на ходу).
- [ ] **Доставка звука на Android** — по итоговой архитектуре Jarvis-клиента (рендер на хосте → отдача, или
      генерация на телефоне). Согласовать с планом Jarvis.
- [ ] **Микс с речью Jarvis:** голос поверх ландшафта (следить за разборчивостью — маскировка).

**Проверка:** голосом «Jarvis, запусти протокол быстрого засыпания» → в наушниках играет корректный
протокол; «Jarvis, стоп» — мгновенно гасит; всё под лимитом громкости и с дисклеймером при первом запуске.

## Фаза 4 — Режим B: акустический CLI / уведомления (опционально) 🔲

> Отдельный трек; делать только если владелец подтвердит ценность (см. развилку 1). Требует обучения
> пользователя «языку» звуков.

- [ ] Набор **~10–20 earcons** для событий KLAS/Jarvis: готово / ошибка / думаю… / долгий процесс
      (сонификация прогресса) / новое уведомление / модель загрузилась (research #2, §1.1).
- [ ] Пара **сонификаций** статуса (напр. загрузка GPU/прогресс как «текстура»/темп).
- [ ] (Задел) **spatial-кодирование** приоритета/канала через `PannerNode` — далёкий горизонт.
- [ ] Документ-«словарь» earcons (что какой звук значит) — для обучения пользователя.

**Проверка:** Jarvis/KLAS отдаёт короткий earcon на реальное событие; владелец различает 5+ токенов на слух.

## Зависимости и риски

- **Главная зависимость:** базовый Jarvis (Фаза 3+). До него — только Фазы 0–2 (песочница на хосте).
- **Лицензии:** SBaGenX/Gnaural — GPL; при заимствовании кода — конфликт с MIT KLAS. Митигация: свой
  генератор на Web Audio (MIT-совместимо) или внешний бинарь процессом.
- **Риск завышенных ожиданий:** entrainment ≠ гарантированный переключатель состояния. Митигация —
  честный тон, позиционирование как аудио-UX, метрики удовлетворённости, не «эффективности лечения».
- **Безопасность:** эпилепсия/musicogenic, громкость, контекст (за рулём/на ходу). Митигация — слой
  безопасности с Фазы 1, дисклеймеры, контекст-чеки.
