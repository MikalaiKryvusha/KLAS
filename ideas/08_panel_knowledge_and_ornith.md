## 1. Скачать, настроить и протестировать Ornith-1.0-35B

Взять в пул модель **Ornith-1.0-35B**, настроить оптимальным образом и протестировать на нашем
харнессе (скорость + ум) — она поднялась в лидерборде свежих моделей (`researches/05`,
`tools/leaderboard.mjs`; SWE-bench 75.6, июнь 2026).

## 2. Пульт управления (Фаза 5, быстро) — единый вход через homepage

Свести управление KLAS в **один вход — homepage** (docker-сервис уже есть): с него — ссылки на
встроенный веб-UI llama-swap (модели, загрузка/выгрузка, метрики), на базу знаний kiwix (`/wiki`),
и статус сервисов. Лёгкая, но очень полезная штука: одна точка, откуда видно и управляемо всё.

## 3. База знаний для LLM: .zim-скрипт + адаптер поиска по википедии

**3a. Опциональный скрипт базы знаний.** Отдельный **опциональный** скрипт (запускает пользователь
по желанию) для скачивания и подключения `.zim`-библиотек (локальная википедия и любые другие базы —
их можно скачивать сколько угодно). `.zim` безумно тяжёлые, поэтому в репозиторий и в общий деплой
не входят — только по явному запросу пользователя.

**3b. Адаптер-«мидлвар», обучающий LLM искать в локальной базе знаний.** Нужен слой-адаптер
(middleware), который «учит и объясняет» связке Zoo Code ↔ llama-swap ↔ llama-server — как локальным
LLM, работающим внутри, **искать нужную информацию в неограниченной быстрой локальной википедии**
(`.zim`). То есть дать локальному агенту инструмент поиска знаний, аналогичный веб-поиску, но по
оффлайн-базе.

Скорее всего, готовые инструменты уже есть — возможно, локальный **MCP-сервер** специально под связку
«llama.cpp ↔ .zim» (или под kiwix). Сначала ищем готовое (гуглёжкой, по README), заимствуем и
интегрируем; если подходящего нет — собираем минимальный свой MCP-сервер поверх поискового API kiwix.
