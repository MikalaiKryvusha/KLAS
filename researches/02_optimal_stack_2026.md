# Research 02 — Оптимальный стек KLAS на июль 2026 (модели ≤16 GB VRAM · движок · фронтенд)

> **Вопрос:** лучший локальный LLM-стек для KLAS (Windows 11, RTX 5070 Ti 16 GB Blackwell/CUDA 13,
> 32 GB RAM): модель (агентные задачи + код + русский), движок (стабильность, OpenAI API, «спит,
> пока не позовут»), агентский VS Code-фронтенд. Приоритеты: **стабильность > ум > скорость**.
>
> **Метод:** deep-research (2026-07-03): 5 поисковых углов → 22 источника → 107 извлечённых фактов.

> верификационную фазу) — факты ниже сырые, при принятии решений проверять по источнику;

## Главные выводы (синтез)

1. **Класс-победитель по железу — MoE ~30B-A3B**: 30B параметров при ~3B активных. Целиком в
   16 GB VRAM в 3–4-битных UD-квантах и при этом быстрые (~35–138 t/s) и умные (SWE-bench ~73%).
2. **Движок оставляем llama.cpp/llama-server**, но: для Blackwell sm_120 под Windows нужен
   свежий релиз/сборка (грабли CUDA 12↔13 документированы); vLLM дисквалифицирован (нет нативного
   Windows); Ollama проще, но меньше контроля (и он уже наследие в нашем окружении).
3. **«Спит, пока не позовут» решается готовым инструментом — [llama-swap](https://github.com/mostlygeek/llama-swap):**
   один Go-бинарник (winget), OpenAI-эндпоинты, автоЗАгрузка модели по полю `model` запроса и
   автоВЫгрузка по `ttl` простоя, работает поверх llama-server. Прямое попадание в Фазу 4 KLAS.
4. **Критично для бага 01:** llama-server при переполнении контекста НЕ падает — возвращает
   **HTTP 400** «request exceeds the available context size» (issue #17284). Значит «ошибка
   переполнения контекста» в Zoo Code — это, вероятно, наш `-c 256000` (KV-кеш такого размера
   физически не влезает в 16 GB) + клиент, не умеющий обрабатывать 400.
5. **Фронтенд:** Cline — «самый безопасный дефолт 2026» (Apache-2.0, ~58.8k звёзд, 5M+ установок,
   любой OpenAI-совместимый API). ⚠️ Противоречие в источниках: один утверждает, что Roo Code
   закрыт и архивирован в мае 2026, другой (старее) описывает его живым — проверить на GitHub.
   Zoo Code в серьёзных обзорах 2026 не фигурирует вовсе.
6. **Русский язык:** свежих сравнений открытых моделей мало (главный community-рейтинг устарел на
   2 года). Qwen-семейство «уверенно работает с кириллицей» (119 языков в токенизаторе). Учитывать:
   русский текст = **1.5–2× больше токенов** → реальный контекст для русских сессий вдвое меньше.

## Движок и менеджмент моделей

- **llama.cpp/llama-server** — база: CUDA, OpenAI API, `--jinja` ОБЯЗАТЕЛЕН для tool calling.
  Грабли Blackwell/Windows: default-сборки могли не включать sm_120 (нужен свежий релиз или
  `CMAKE_CUDA_ARCHITECTURES=120`); конфликт cudart64_12.dll при параллельной установке CUDA 12.x;
  переменная `CUDA_VISIBLE_DEVICES=-1` от NVIDIA-утилит скрывает GPU.
- **llama-swap** поверх llama-server: автозагрузка/автовыгрузка (ttl), несколько моделей через
  `matrix` (например, кодовая + эмбеддинги), winget, Windows поддержан. → кандидат в ядро KLAS.
- **vLLM** — только WSL2, мимо. **LM Studio** — проприетарный, но реально работает на 5070 Ti
  (замеры выше); годится как ручной испытательный стенд, не как сервис.

## Рекомендуемые параметры llama-server (из практики агентного кодинга, январь 2026)

```
llama-server -m <model>.gguf -c <РЕАЛИСТИЧНЫЙ ctx под VRAM> -ngl 99 -fa --jinja \
  -ctk q8_0 -ctv q8_0   # KV q8_0; q4_0 портит многофайловую агентику
```
Плюс сэмплинг для Qwen3-Coder: temp 0.7, top_p 0.8, top_k 20, rep.penalty 1.05.
KV-бюджет: ~+25% к весам на 8K ctx, ~+100% на 32K (для плотных); наш `-c 256000` был фантазией.

## Фронтенды VS Code

- **Cline** — дефолт: крупнейшее сообщество, OpenAI-совместимый/Ollama/LM Studio, локальные данные.
  Нюанс: Cline/Roo/Kilo используют СВОЙ текстовый формат вызова инструментов (не нативный tool
  calling API) — надёжность зависит от модели, quirks шаблонов GGUF влияют меньше.
- **Roo Code** — ✅ проверено через GitHub API (2026-07-03): репозиторий **архивирован**,
  последний push 2026-05-15 — из кандидатов исключён.
- **Continue** — зрелый, но проект пивотнулся (IDE-опыт может стагнировать). **Kilo Code** — молодой форк.
- **Zoo Code** (наш текущий) — по информации владельца, форк Cline с хорошими отзывами. Значит,
  поведение с локальными моделями (свой текстовый формат tool calling, работа через
  OpenAI-совместимый API) наследуется от Cline — менять фронтенд не обязательно; сравнить
  Zoo Code vs Cline на нашем харнессе при случае.

## Связки-кандидаты

1. **Основная (ум+скорость, всё в VRAM):** llama-server (свежий CUDA-13 релиз) + **Qwen3.5/3.6-35B-A3B
   UD-IQ3_S** + llama-swap + **Cline**. Ожидание: ~100+ t/s, 50–100K ctx, сильная агентика.
2. **Агентный кодинг:** llama-server + **Qwen3-Coder-30B-A3B UD-Q5_K_XL** (KV в q8_0, ctx ~48–64K,
   часть KV возможно в RAM) + llama-swap + Cline. Проверить поведение >30K ctx на свежем билде.
3. **Консервативная (простота/стабильность):** llama-server + **GPT-OSS 20B** (ctx ≤60K) или
   **Devstral Small 24B Q4_K_M** + llama-swap + Cline.

**Выводы v2 (2026-07-07):**
- **Де-сатурация удалась:** разброс 6–7/9 вместо однородных 6/6. Но главная ценность — **профили по
  задачам**, а не общий балл (модели валят РАЗНОЕ). N=9, один прогон, temp 0.1 → не абсолютный рейтинг.
- **multistep-math (14 ТБ × 60% = 8.4) валят ВСЕ 5.** Многошаговый счёт с десятичным процентом —
  системная слабость этого класса локальных моделей. Учитывать: не полагаться на них в арифметике «в уме».
- **long-ctx-synth (две иголки → сумма):** прошли только **Qwythos-9B** (обучен на 1M!) и **Qwen3.6-27B**;
  gemma/qwen3.5/ornith — нет. Прямое подтверждение силы Qwythos на длинном контексте (крик владельца
  «мало памяти») и слабость «умного» qwen3.5 именно на СИНТЕЗЕ по контексту (одну иголку берёт, две — нет).
- **json-sort (сортировка по полю):** прошли gemma и qwen3.6; qwythos/qwen3.5/ornith — нет.
- **gemma-4-12b неожиданно ровная и быстрая (7/9)** — хорошо сбалансирована, холодный старт ~23 с.
- **ornith-35b подтверждает непригодность в дейли:** long-ctx = 106 с (медленный prompt-processing),
  всё медленнее в разы, при этом 6/9 — не окупается.
- **Нет доминирующей модели:** лидеры (gemma, Qwythos) сильны в РАЗНОМ (gemma — инструкции/JSON;
  Qwythos — длинный контекст). Выбор основной = компромисс под задачу (ум × контекст × скорость).

**Идеи для v3 (когда понадобится тоньше):** многошаговая математика с проверкой рассуждения; цепочки из
3+ tool-call с ветвлением; многофайловая правка кода; удержание инструкции на длинной сессии; более
хитрые дистракторы в long-context. multistep-math и json-sort — оставить как «кислотный тест».
