9. Рефлексируй по багам в bugs/     # один md на баг; по BUG_FIXING_FRAMEWORK.md
10. Периодически перечитывай ключевые руководящие документы:
    - PHILOSOPHY.md   ← принцип простоты; застрял — сначала сюда

    - BUG_FIXING_FRAMEWORK.md

11. Рассказывай в чате, хотя бы немного, естественным языком — что делаешь прямо сейчас, — чтобы
    человек мог мельком глянуть и следить за ходом.
12. Документы от человека (идеи, баги, фичи): читай их, исправляй опечатки, минимально
    реструктурируй в чистый структурированный формат для потребления ИИ. После реализации по такому
    документу впиши в него статус и дату реализации.

## Цель проекта

**KLAS (Krinik Local Agent System)** — self-hosted экосистема агентского ИИ на личном Windows 11 ПК
владельца (RTX 5070 Ti 16 GB, 32 GB RAM, Ryzen 5700G): локальная LLM на геймерской GPU, автономные
ИИ-агенты, веб-дашборд управления и база знаний в виде локальной википедии — полная приватность
данных и оффлайн-эффективность. Система «спит», пока не позвали, работает стабильно (без ошибок
переполнения контекста) и максимально приближается по уму и удобству к Claude AI (Fable 5) в пределах
локального железа. Служит владельцу и его близким. Полное видение — `GOAL.md`.

## Архитектура — карта

Проект — это **инженерия локальной ИИ-инфраструктуры**: сам код здесь минимален (JS: .mjs + Node —
для скриптов, инструментов, бекенда и БД), главное — конфигурации, скрипты, исследования и документы.
Целевой дом системы — `F:\KLAS\` (корень = git-репозиторий с документами KAIF + инфраструктурные
поддиректории):

```
F:\KLAS\
├── (документы KAIF/проекта, tools\, plans\, bugs\, …)  ← управляющий центр (git)
├── llamacpp\              ← движок llama.cpp (CUDA-сборка: llama-server.exe, llama-bench.exe, …)
│   └── bat\gemma4-12b.bat ← текущий профиль запуска LLM-сервера
├── LLMs\LLAMACPP_MODELS\  ← GGUF-модели (gemma-4-12b Q4_K_XL ~7.4GB; Qwen3.6-27B Q4_K_XL ~17.9GB)
├── homepage\ caddy\ kiwixdb\  ← docker-сервисы (см. docker-compose.yml); kiwix = база знаний (википедия)
├── docker-compose.yml     ← kiwix (:8080!), homepage (:3005), caddy (:80/:443)
├── mcp\                   ← MCP-серверы (web-search-mcp)
└── nssm\ tailscale_funnel_443.bat ← сервис-менеджер и внешний доступ через Tailscale
```

**ПРАВИЛО:** документы KAIF (корень) — управляющий центр: он документирует, конфигурирует и проверяет
инфраструктурные директории, но меняет их только осознанно, точечно и с фиксацией в документах. Один
источник правды по портам и профилям запуска — карты этого проекта.

Полная карта файлов — `PROJECT_STRUCTURE_EXTERNAL_MAP.md`; логическая архитектура —
`PROJECT_ARCHITECTURE_INTERNAL_MAP.md`.

## «Сборка» (запуск/проверка стека)

Классической сборки кода нет. «Сборка» здесь = **запуск LLM-стека и проверка, что он отвечает**:

```powershell
# запустить текущий профиль LLM-сервера (окно останется открытым):
F:\KLAS\llamacpp\bat\gemma4-12b.bat

# проверить, что сервер жив (порт по умолчанию llama-server — 8080):
curl http://127.0.0.1:8080/health
```

Особенности окружения:
- ⚠️ Порт 8080 также маппится docker-контейнером kiwix (`docker-compose.yml`) — возможен конфликт;
  всегда проверяй, кто слушает порт: `netstat -ano | findstr :8080`.
- GPU: RTX 5070 Ti **16 GB VRAM** — бюджет для модель+KV-кеш. Проверка занятости: `nvidia-smi`.
- Не запускай второй llama-server поверх работающего — сначала проверь `/health` и процессы.

## Тестовый харнесс (как агент наблюдает и управляет системой)

Харнесс — важнейшая инвестиция для автономной работы. Текущий минимум — HTTP API llama-server и
стандартные утилиты; наращивай его скриптами в `tools/` и документируй здесь новые команды.

| Команда | Что делает |
|---------|------------|
| `curl http://127.0.0.1:8080/health` | Жив ли LLM-сервер |
| `curl http://127.0.0.1:8080/v1/models` | Какая модель загружена |
| `curl http://127.0.0.1:8080/props` | Реальные параметры сервера (контекст, слоты и т.д.) |
| `curl -s http://127.0.0.1:8080/v1/chat/completions -H "Content-Type: application/json" -d '{...}'` | Тестовый запрос к модели (OpenAI-совместимый API) |
| `F:\KLAS\llamacpp\llama-bench.exe -m <model.gguf>` | Бенчмарк скорости (t/s) модели |
| `nvidia-smi` | Занятость VRAM/GPU |
| `netstat -ano \| findstr :<port>` | Кто слушает порт |
| `tasklist \| findstr llama` | Запущенные процессы llama |

> Правило: сделал что-то руками для проверки — добавь скрипт/команду, чтобы в следующий раз это был
> один шаг, и допиши строку в эту таблицу.

## Беклог и DONE-тег

Чтобы по одному списку файлов было видно, что открыто, а что закрыто, — **вставляй слово `DONE` в имя
файла после номера, когда задача файла завершена и проверена:**

```
bugs/04_modal.md           →  bugs/04_DONE_modal.md
ideas/07_dev_menu.md       →  ideas/07_DONE_dev_menu.md
```

**Правило (каждый раз при работе с файлами багов/идей):**
- Закончил баг/идею и это ПОДТВЕРЖДЁННО закрыто (статус ✅, проверено) — сразу переименуй, вставив
  `DONE` после номера: `git mv <NN>_<name>.md <NN>_DONE_<name>.md`.
- Файл в работе / частично / только исследование — НЕ помечай `DONE` (🔧/🟡/🔬 = ещё не готово).
- Используй `git mv` (сохраняет историю). Номер не меняй.
- Справочные документы (master plan, карты и т.п.) — НЕ задачи, никогда не помечаются DONE.

**Скилл ревизии беклога — `/check-backlog`:** проходит `bugs/`, `plans/`, `ideas/`, собирает всё без
`DONE` как открытый беклог и помечает действительно закрытые файлы DONE (с добавлением статус-секции).

**Скилл репорта багов — `/report-bug`:** встретил дефект при работе/тестировании — заведи отдельный md
в `bugs/` по канону, по `BUG_FIXING_FRAMEWORK.md`. Агент ведёт собственный беклог багов — один документ
на дефект, ничего не теряется.

**Скилл предложения идей — `/propose-idea`:** появилась стоящая идея, вписывающаяся в мастер-план и
видение человека, — оформи её как md в `ideas/` со статусом «❓ ждёт одобрения человека». Идея агента —
вклад в ВИДЕНИЕ продукта → реализуй ТОЛЬКО после одобрения человека.

## Решения, которые агент НЕ принимает в одиночку — интервью

Перед значимой новой фичей и при любой развилке бренд/UX/архитектура проводи **интервью** с человеком
через скилл `/interview`: закрытые вопросы A/B/C с рекомендацией первым пунктом; человек отвечает прямо
в `interviews/interview_NNN_<topic>.md`. Никогда не принимай решения по UI/UX/бренду/архитектуре без
подтверждения. Всё остальное — решай сам с разумными дефолтами и отчитывайся в чате.

Эмпирическое правило: *дёшево ли это откатить?* Да — решай сам. Формирует бренд/архитектуру/UX
надолго — интервью.

Для этого проекта владельческие развилки — это, например: замена ключевого элемента стека (движок LLM,
основная модель, агентский фронтенд), форма веб-дашборда, открытие внешнего доступа к серверу.
