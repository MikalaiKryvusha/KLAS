# /autoloop — длинная автономная работа по пулу задач

Человек отошёл / не хочет вмешиваться. Задача агента: **войти в самоуправляемый цикл и молотить пул
АВТОНОМНЫХ задач** одну за другой, пока пул не опустеет или не появится блокер. Между задачами —
короткие отчёты в чат, чтобы человек мог заглянуть и увидеть прогресс.

## 🔁 Самовызов и самоперезапуск (агент запускает себя сам)

Агент может и ДОЛЖЕН вызывать этот скилл **по собственной инициативе** (через Skill tool), не дожидаясь
явной команды, когда выполняются ВСЕ условия:
- в АВТОНОМНОМ пуле есть открытые задачи (`STATUS.md` → «🤖 Пул автономного беклога»);
- прямо сейчас нет активной интерактивной задачи/вопроса от человека (он отошёл / дал автономию);
- окружение готово (стек запускается; нужные сервисы доступны).

Более того, **цикл самоперезапускается**: закончив одну задачу (коммит + короткий отчёт), агент
переходит к следующей итерации этого же скилла — **немедленно снова вызывает `/autoloop`** и берёт
следующую задачу, не дожидаясь команды. Так серия идёт долго и непрерывно. Стоп — ТОЛЬКО по условиям
из «Когда ОСТАНОВИТЬ» ниже.

> ⛔ **АНТИПАТТЕРН (так не делать):** закончить задачу и «спросить человека: продолжать или стоп?»,
> когда в пуле ЕЩЁ есть задача. Это нарушение скилла. «Длинная сессия», «поздно», «много контекста»,
> «риск лимита», «устал» — НЕ стоп-условия. Есть автономная задача в пуле — сам вызывай `/autoloop` и
> продолжай. Останавливайся ИСКЛЮЧИТЕЛЬНО по списку ниже. Контекст и лимиты — забота харнесса, а не
> причина рвать автономную серию.

> Не злоупотребляй: самовызов — для автономного пула. Если человек дал конкретную интерактивную
> задачу — делай её, не убегай в цикл. Если он явно сказал «стоп / хватит / пауза» — остановись.

> 🧠 Работай по `PHILOSOPHY.md` (ПРОСТО, KISS, Оккам) и `BUG_FIXING_FRAMEWORK.md` (3 попытки → стоп →
> `/bug-research`). Комментируй код. Рассказывай в чате, что делаешь.

## Граница автономии (жёсткий лимит)

Некоторым задачам нужен человек (его руки, глаза, аккаунты, решения по UX/бренду/архитектуре) или
особые ресурсы, недоступные агенту прямо сейчас. **Такие задачи НЕ брать — отложи, возьми другую.**
Держи границу явной в пуле `STATUS.md`: там только задачи, проверяемые БЕЗ человека и без таких
ресурсов.

## Шаг 0. Подготовка (один раз в начале)

1. Прочитай: `STATUS.md` (секция «🤖 Пул автономного беклога»), `PHILOSOPHY.md`, `AGENT_GUIDE.md`,
   `BUG_FIXING_FRAMEWORK.md`, релевантные `ideas/*` и `bugs/*`.
2. Проверь готовность окружения (стек, порты, VRAM — см. `AGENT_GUIDE.md`, «Тестовый харнесс»).
3. Собери/освежи рабочий список пула из STATUS. Скажи человеку одним абзацем: что в пуле, с какой
   задачи начинаешь и почему.

## Цикл (повторять на каждую задачу)

1. **Выбери** следующую автономную задачу из пула (приоритет из STATUS). Убедись, что её можно
   проверить БЕЗ человека/особых ресурсов. Нельзя — отложи, возьми следующую.
2. **Пойми её просто** (PHILOSOPHY) — сформулируй в 1–2 предложениях. Для багов — открой/создай
   документ в `bugs/`.
3. **Реализуй** точечно, с комментариями. Не усложняй.
4. **Запусти/проверь стек** (например `F:\KLAS\llamacpp\bat\gemma4-12b.bat`, затем
   `curl http://127.0.0.1:8080/health`). Ошибки — чини, не коммить сломанное состояние.
5. **Разверни/запусти**, как требует задача.
6. **Проверь автономно** на харнессе (таблица в `AGENT_GUIDE.md`: `/health`, `/props`, `nvidia-smi`,
   `llama-bench`, `netstat`). Смотри на результат внимательно — не выдавай желаемое за действительное;
   проверяй объективно.
7. **Цикл фикса** на баге: фикс → запуск → тест → логи (свежие по таймстемпу). Правило **3 попыток** →
   `/bug-research` (без кода) → потом фикс.
8. **Зафиксируй знание**: по багам — рефлексия в `bugs/NN_*.md`; по фичам — статус/дата в `ideas/*`;
   обнови `STATUS.md`.
9. **Коммит** — маленький, чтобы не терять прогресс: `git add -A && git commit -m "..."` (стиль из
   `AGENT_GUIDE.md`, с трейлером Co-Authored-By).
10. **Короткий отчёт в чат** (1–3 строки): что сделал, что проверил, что дальше. → следующая задача.

## Само-темп (чтобы цикл шёл ДОЛГО)

- Иди задача за задачей без остановок на подтверждения (кроме деструктивных задач).
- Ждёшь фоновую операцию (долгая загрузка модели) — продолжай, когда готово; не пингуй человека.
- Нужно «продолжить по таймеру» — используй механизм цикла харнесса (`ScheduleWakeup`/`/loop`) с
  разумным интервалом, передавая этот же скилл, чтобы цикл возобновился.

## Когда ОСТАНОВИТЬ цикл (и отчитаться человеку)

- Автономный пул исчерпан (всё оставшееся требует человека/ресурсов).
- Серьёзная развилка UI/UX/бренд/архитектура, которую агент НЕ решает один → заведи `/interview` и пауза.
- Что-то деструктивное/необратимое (релиз, удаление, force-push) — не делай один, спроси.

В конце цикла — итог: что сделано за серию (список коммитов), что отложено и почему, что предложить
дальше. Коммиты по пути гарантируют: прогресс не теряется.

## Заметки

- НЕ публикуй релизы, НЕ делай force-push, НЕ удаляй чужую работу — это вне автономии.
- Держи дерево чистым перед коммитом; тяжёлые артефакты — в `.gitignore`.
- Читай только СВЕЖИЕ логи (проверяй таймстемп).
