# /bug-research — глубокое исследование бага без кодинга

Используется, когда баг **не поддаётся прямым попыткам** (правило: после **3 неудачных итераций**
фикс→запуск→тест прекращаем идти вслепую — см. `BUG_FIXING_FRAMEWORK.md`). Случайное тыканье тратит
время и запуски; остановись и ПОЙМИ причину.

> ⛔ В этом скилле мы НЕ пишем код, НЕ фиксим, НЕ пересобираем, НЕ запускаем софт. Только чтение,
> поиск, анализ, рефлексия и запись в md-документ бага. Чистая когнитивная работа.

## Шаг 0. Заякорись на баге

- Открой документ бага в `bugs/NN_*.md` (нет — создай по `BUG_FIXING_FRAMEWORK.md`).
- Коротко выпиши: симптом, что пробовали (журнал попыток), при каких условиях воспроизводится.
- Скажи человеку одной строкой, что переключаешься в режим исследования (хватит тыкать вслепую).

## Шаг 1. Веб-поиск — собери СЫРУЮ базу знаний

Сделай несколько прицельных запросов (`WebSearch`), затем вытяни самые релевантные страницы
(`WebFetch`). Смотри: GitHub issues/discussions/wiki библиотеки (llama.cpp, Zoo Code и т.д.),
Stack Overflow, Reddit (r/LocalLLaMA), официальную документацию.

- Формулируй запросы по точным API/параметрам/симптомам (имена флагов, тексты ошибок, версии).
- **Записывай сырые данные ДОСЛОВНО** в документ бага в секцию «База знаний — сырые данные поиска»:
  цитаты мейнтейнеров, сигнатуры/флаги, объяснения, ссылки на источники. Это знание для будущих
  сессий — не пересказывай вольно; сохраняй факты и ссылки.
- Отдельно пометь: **возможно ли вообще то, что мы делаем** (иногда это ограничение
  платформы/библиотеки/модели).

## Шаг 2. Анализ кода/конфигов — найди, где причина (без правок)

Читай и трассируй цепочку, связанную с багом (данные/вызовы/состояние). Не правь — препарируй.

- Построй цепочку (поток данных / поток вызовов) от источника проблемы до симптома; запиши её.
- Найди подозрительные места: кто что передаёт, где значение теряется/искажается, какие допущения
  сделаны (например: настройки Zoo Code ↔ параметры `gemma4-12b.bat` ↔ лимиты модели).
- Сопоставь наши попытки с узнанным («попытка → что она делает по документации → почему не помогла»).

> 🧠 Держи в уме `PHILOSOPHY.md`: застревание обычно значит, что РЕШЕНИЕ слишком сложное из-за
> непонимания задачи, а не что задача сложная. Ищи ПРОСТОЙ поддерживаемый путь (KISS + Оккам). Если
> гипотеза/план разбухают — ты, вероятно, всё ещё не понял задачу; переформулируй её простыми словами.

## Шаг 3. Рефлексия и гипотезы

В документе бага сформулируй:
- **Гипотезу корневой причины** (одну-две, обоснованные шагами 1–2).
- **Следующие шаги для прицельной кодинг-сессии** — конкретные, проверяемые (какие файлы, какой
  эксперимент, как надёжно проверить результат — не «на глазок»).
- **Открытые вопросы человеку**, если выбор подхода за ним.

## Шаг 4. Резюме в чат

Кратко: что нашёл (ключевые факты), рабочая гипотеза причины, предлагаемый план следующего кодинг-
захода. НЕ начинай фиксить внутри этого скилла — он заканчивается готовой базой знаний.

## Заметки
- Надёжный метод проверки важнее скорости: если визуальная проверка ненадёжна — придумай объективную
  (контрольные входы, измерения, логи размеров) и запиши в документ.
- Цель скилла: превратить «не работает, тыкаю вслепую» в «понимаю причину, есть план».
