# /propose-idea — оформить идею в ideas/ (на одобрение человека)

Когда у агента появляется стоящая идея (фича, улучшение UX/архитектуры, инструмент), он оформляет её
ОТДЕЛЬНЫМ md в `ideas/` по тем же правилам, что и существующие идеи. **Идея агента — вклад в ВИДЕНИЕ
продукта, а видение принадлежит владельцу (человеку).** Поэтому такая идея создаётся со статусом
«ждёт одобрения» и **НЕ реализуется, пока человек не одобрит**.

## Когда вызывать

- Появилась идея, продвигающая проект к целям `MASTER_PLAN.md`/`PHILOSOPHY.md` и не противоречащая
  видению человека.
- НЕ для мелких технических решений внутри уже одобренной задачи (решай сам и просто делай).
- НЕ для вопросов, требующих ВВОДА человека по уже идущей работе — это `/interview` (вопрос), не идея.
  Разница: `/interview` = «мне нужен твой ответ, чтобы продолжить»; `/propose-idea` = «предлагаю НОВОЕ
  направление, оцени».

## Что делать

1. **Сверь с видением ДО оформления.** Прочитай (или вспомни) `MASTER_PLAN.md`, `PHILOSOPHY.md`
   (KISS) и решения в `interviews/`. Идея должна согласовываться. Если противоречит / уводит в
   сторону / усложняет без нужды — не предлагай (или переформулируй проще).

2. **Определи следующий номер.** `ls ideas/` → максимальный `NN` + 1. Имя:
   `ideas/NN_<short_english_name>.md` (как у соседей; БЕЗ тега `DONE`).

3. **Напиши документ по канону** (структура как у существующих идей):
   ```
   # Idea NN — <название>

   > Источник: ИИ-агент (dayloop/nightloop/наблюдение), <дата>.
   > Статус: ❓ ЖДЁТ ОДОБРЕНИЯ ЧЕЛОВЕКА — идея агента = элемент видения; без одобрения не реализуется.

   ## Суть
   <что предлагается, кратко>

   ## Зачем / какую проблему решает
   <боль/возможность; для кого>

   ## Как вписывается в мастер-план и видение
   <явная привязка к master_plan / принятым решениям — почему это «по курсу», а не вбок>

   ## Набросок реализации (KISS)
   <примерный подход; переиспользование существующего; объём/риски>

   ## Открытые вопросы / что требует ответа человека
   <развилки, которые закрывает владелец>
   ```

4. **Пометь на ревью и НЕ реализуй:**
   - Строка в `STATUS.md`: «❓ ждёт ревью человека: идея NN — <одна строка>».
   - Закоммить документ (`git add -A && git commit -m "docs(ideaNN): proposal — …"`).
   - **НЕ начинай реализацию**, пока человек явно не одобрит. В автоцикле — продолжай ДРУГИЕ задачи.

5. **После реакции человека:**
   - Одобрено → бери в работу (технические решения внутри — твои). После реализации: статус ✅ +
     дата, и по конвенции DONE-тега — `git mv` с тегом `DONE` и статус-секцией внутри.
   - Отклонено/переформулировано → отрази его решение в документе (или удали идею, если отклонена).

## Заметки
- Идея — ПРЕДЛОЖЕНИЕ, а не разрешение действовать. Дисциплина: видение продукта формирует человек.
- Лучше собрать несколько мелких идей осмысленно, чем плодить почти-дубликаты. Проверь существующие
  открытые идеи (`ls ideas/`), чтобы не дублировать.
