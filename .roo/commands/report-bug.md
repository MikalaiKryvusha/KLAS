# /report-bug — завести документ бага в bugs/ (агент ведёт свой беклог багов)

Всякий раз, когда агент пишет код/конфиг, тестирует и **встречает баг** (падение, неверное поведение,
артефакт, регрессия, дефект библиотеки), он заводит ОТДЕЛЬНЫЙ md-документ в `bugs/` по тем же правилам,
что и существующие документы багов. Так агент накапливает себе беклог багов — ничего не теряется, к
любому багу можно вернуться (или передать его `/bug-research`).

> Правило проекта (`AGENT_GUIDE.md`): «по каждому багу, даже мелкому, — рефлексия и фиксация знания в
> отдельном md в `bugs/`». Этот скилл автоматизирует ровно это.

## Когда вызывать

- Агент увидел дефект при разработке/тесте, который сейчас НЕ чинит (или чинит, но хочет зафиксировать
  криминалистику/постмортем).
- Баг нужно отложить (взять другую задачу), не потеряв его.
- Человек просит завести баг.
- НЕ для застревания-от-непонимания (это `PHILOSOPHY.md`) и не вместо починки тривиальной опечатки.

## Что делать

1. **Определи следующий номер.** `ls bugs/` → максимальный двузначный `NN` + 1. Имя файла:
   `bugs/NN_<short_english_name>.md` (snake_case, как у соседей; БЕЗ тега `DONE` — баг открыт).

2. **Собери факты ДО написания** (не выдумывай — документ бага ценен фактами):
   - Симптом: что именно наблюдается (и чем отличается от ожидаемого).
   - Воспроизведение: детерминированные шаги. Где возможно — через харнесс, чтобы баг воспроизводился
     без человека.
   - Криминалистика: логи / падение / измерения — приложи ключевые строки (stack, сообщение об аварии,
     коды ошибок, размеры).
   - Сборка/версия, окружение, режим.

3. **Напиши документ по канону** (структура как у существующих багов):
   ```
   # Bug NN — <описание одной строкой>

   **Статус:** 🔴 OPEN   (или 🟡 частично / 🔬 только исследование / 🔧 фикс ждёт проверки)
   **Версия/сборка:** <...>   ·   **Когда/контекст:** <дата, в ходе какой задачи найден>

   ## Симптом
   <что наблюдается>

   ## Воспроизведение (детерминированное)
   <шаги; команды харнесса, если есть>

   ## Криминалистика
   <ключевые логи / падение / измерения>

   ## Корневая причина / Гипотезы
   <причина, если ясна; иначе ранжированные гипотезы — НЕ патчить вслепую>

   ## План фикса (или сам фикс, если сделан)
   <шаги; связь с архитектурой / другими багами>

   ## Ссылки
   <связанные баги / идеи / интервью>
   ```
   Следуй `BUG_FIXING_FRAMEWORK.md`. Если баг в сторонней библиотеке — заведи для неё тикет (например,
   через `gh`) и сошлись на него из документа.

4. **Зафиксируй в беклоге/процессе:**
   - Если важный/блокирующий — короткая строка в `STATUS.md`.
   - Коммит (в автоциклах — по обычной дисциплине): `git add -A && git commit -m "docs(bugNN): …"`.

5. **Жизненный цикл:** пока открыт — файл БЕЗ `DONE`. Когда ПОДТВЕРЖДЁННО закрыт (починен и
   проверен) — переименуй `git mv bugs/NN_x.md bugs/NN_DONE_x.md` и добавь секцию
   `## ✅ STATUS: DONE (дата)` (что сделано / как проверено). Ревизия беклога — скилл `/check-backlog`.

## Заметки
- Лучше завести баг и оставить открытым, чем потерять. Фактическая точность важнее прозы.
- Баг сопротивляется (≥3 слепых попытки фикса) — переходи к `/bug-research` (исследование без кода)
  на том же документе.
