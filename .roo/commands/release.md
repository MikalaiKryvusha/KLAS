# /release — опубликовать релиз на GitHub

Человек просит выпустить новую версию. Это **необратимое внешнее действие** (публичный тег +
GitHub Release). Выполни рутину **по порядку**; рассказывай каждый шаг в чате. Шаг упал — остановись,
покажи ошибку, НЕ продолжай вслепую.

> ⚠️ **ТРЕБУЕТСЯ ПОДТВЕРЖДЕНИЕ.** Перед самой публикацией покажи человеку: какая версия будет
> (текущая → новая), что дерево чистое, что стек проверен. Публикуй только по его явному «да». Релиз =
> публичный тег и Release, откатывать неприятно. **В автономном режиме (`/autoloop`/циклы) релизы НЕ
> публиковать.**
>
> ℹ️ Пока у проекта нет remote-репозитория (см. `interviews/interview_001`), релиз невозможен —
> сначала закрой интервью и настрой remote.

## Шаг 0. Определи тип бампа

Подтверди с человеком (или подтверди дефолт): patch / minor / major. Назови текущую → новую версию.

## Шаг 1. Предпроверка окружения (не релизь на грязном/сломанном дереве)

```bash
git status --short          # дерево должно быть ЧИСТЫМ (кроме gitignored-артефактов)
git branch --show-current   # релизная ветка (main)
git pull --rebase           # чтобы push был fast-forward
gh auth status              # gh залогинен (нужен для GitHub Release)
```
Дерево грязное — сначала закоммить/разберись (`/pause` или обычный коммит).

## Шаг 2. Освежи README (все языки)

Приведи `README.md` в соответствие с реальностью: статусы фаз, работающие возможности, инструкции.
Не выдумывай — отражай только реально сделанное и проверенное (сверь `STATUS.md` и закрытые
`bugs/`/`ideas/` `*_DONE_*`).

## Шаг 3. Регенерируй артефакты

В этом проекте генерируемых артефактов пока нет — пропусти (появятся — добавь сюда команды).

## Шаг 4. Контрольная проверка (перед релизом)

Прогони харнесс стека: `curl http://127.0.0.1:8080/health` (если релиз касается стека — полный прогон
таблицы харнесса из `AGENT_GUIDE.md`). Это ловит проблемы ДО бампа версии, чтобы не оставить
полувыпущенную версию.

## Шаг 5. Закоммить изменения документов (до релиза)

Закоммить обновления README/документов, чтобы коммит `release: X.Y` был чистым бампом версии:
```bash
git add -A && git commit -m "docs: README for release X.Y"
```

## Шаг 6. Публикация (после подтверждения человека)

```bash
# бамп версии в package.json ("version"), затем:
git commit -am "release: X.Y" && git tag vX.Y && git push && git push --tags
gh release create vX.Y --title "vX.Y" --generate-notes
```

## Шаг 7. Проверь и отчитайся

```bash
gh release view vX.Y        # релиз существует
git log --oneline -3        # релизный коммит + тег видны
```
Отчитайся человеку: версия, ссылка на релиз, что приложено. Готово.

## Заметки
- Релизы бампают minor/major; обычные рабочие коммиты — build/patch.
- Push отклонён (non-fast-forward) — `git pull --rebase` и повтори. На шаге 6 это критично: тег мог
  уже создаться локально — проверь `git tag` и `git tag -d vX.Y` перед повтором.
- НИКОГДА не force-push и не удаляй чужие теги/релизы. Что-то пошло не так при публикации —
  остановись и покажи человеку, не «чини» вслепую.
- Не релизь в автономном режиме — только по явной просьбе человека.
