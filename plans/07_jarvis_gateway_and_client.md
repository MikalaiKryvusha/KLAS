## Стержневая идея (что именно строим)

**Тонкий «Jarvis Gateway»** на хосте + **лёгкий Android-клиент** на телефоне, связанные **через уже
развёрнутый в KLAS Tailscale**. НЕ «обширный кастомный API» — а маленькое стандартное ядро (realtime-
сессия голоса + control-API), переиспользующее существующие llama-server `/v1` и Tailscale. Обширным
шлюз становится органически, по мере добавления навыков.

**Принципы (PHILOSOPHY.md + заметки владельца):**
- KISS: не выставлять Jarvis в публичный интернет; не плодить эндпоинты; копировать форму сложившихся
  стандартов (OpenAI Realtime / Wyoming), а не изобретать.
- Минимум сущностей: транспорт — Tailscale (есть), LLM — llama-server (есть). Дописать только голос-сессию,
  STT, TTS, роутер навыков.
- Приватность — стержень: голос/транскрипты/память не покидают хост; канал только в tailnet.
- Бери лучшее: Whisper.cpp, Piper, Pipecat/LiveKit, HA Companion App — как база/референс.

## Развилки для `/interview` ПЕРЕД реализацией

1. **Android-клиент:** свой с нуля или база на Home Assistant Companion App? (рекомендую сначала оценить форк HA)
2. **Оркестрация на хосте:** Node (ближе к KLAS) vs Python/Pipecat (зрелее для realtime-аудио). Рекомендую гибрид: control-API на Node, realtime-аудио пайплайн на Python.
3. **Транспорт:** начать с WebSocket (прототип в tailnet) → мигрировать на WebRTC (мобильный прод). Рекомендую поэтапно.
4. **STT/TTS старт:** Whisper.cpp + Piper (голоса Вихрова/Joi обучить позже). Ок?
5. **Wake word:** on-device (microWakeWord/openWakeWord) + запасная кнопка на гарнитуре. Ок?

## Фаза 1 — «Скелет разговора» на хосте (без телефона, без сети) 🔲

**Цель:** на самом хосте собрать каскад STT→LLM→TTS с одним локальным клиентом (микрофон/динамики ПК),
доказать пайплайн и качество, ничего не завязывая на мобильность.

- [ ] **STT:** Whisper.cpp / faster-whisper, потоковое распознавание (частичные результаты).
- [ ] **LLM:** переиспользовать llama-server `/v1` (уже есть) — интент/ответ; системный промпт персоны.
- [ ] **TTS:** Piper (быстрый neural TTS) — временный голос, пока не обучены Вихров/Joi.
- [ ] **Оркестратор сессии** (Node или Python/Pipecat — по развилке 2): склейка STT→LLM→TTS со **стримингом**
      («начал говорить, пока думаю») и **barge-in** (перебивание голосом).
- [ ] **Харнесс:** скрипт замера сквозной задержки (микрофон→первый звук ответа) + строка в таблицу
      `AGENT_GUIDE.md`.

**Проверка:** сказать в микрофон ПК фразу → Jarvis отвечает голосом за <~1–2 с до первого слова; перебивание
работает.

## Фаза 2 — Jarvis Gateway: тонкий API (2 поверхности) 🔲

**Цель:** вынести пайплайн за сетевую границу — оформить как сервис с маленьким стандартным API.

- [ ] **Realtime Voice Session (WebSocket)** — событийный протокол (по образу OpenAI Realtime / Wyoming):
      `session.start(персона, язык)`, `audio.chunk`, `barge_in`, `session.stop` (клиент→хост);
      `stt.partial/final`, `assistant.text.delta`, `audio.chunk`(TTS), `skill.invoked`, `error` (хост→клиент).
      Одна долгоживущая stateful-сессия = один разговор.
- [ ] **Control API (маленький REST/JSON):** выбор персоны **Jarvis/Joi**, конфиг голоса, навыки (CRUD из
      idea 05), доступ к памяти, запуск PAI-протоколов ([idea PAI](../ideas/psycho_acoustic_interface/)).
- [ ] **Тестовый веб-клиент** (страница в пульте/homepage): микрофон браузера → WS → ответ голосом. Проверка
      Gateway без Android.

**Проверка:** из браузера на другом ПК в tailnet — полноценный голосовой разговор с Jarvis через Gateway.

## Фаза 3 — Транспорт через Tailscale (защищённый канал устройство↔хост) 🔲

- [ ] Android-устройство Криника — узел **tailnet** (WireGuard, обход NAT, без публичного IP).
- [ ] Gateway слушает только на tailnet-интерфейсе (НЕ публичный funnel — приватность голоса).
- [ ] Аутентификация: узел tailnet + Bearer-ключ на control-API (практика ключей KLAS уже есть).
- [ ] Проверка устойчивости канала на сотовой сети (потери/джиттер) — метрика для решения о WebRTC (Фаза 5).

**Проверка:** веб-клиент с телефона в сотовой сети через tailnet разговаривает с Jarvis дома.

## Фаза 4 — Android-клиент (носимый Jarvis/Joi) 🔲

> Самый «мобильно-сложный» слой (researches/07 §4). Решить развилку 1 (свой vs форк HA Companion App).

- [ ] **On-device wake word** «Jarvis»/«Joi» (microWakeWord/openWakeWord) — сессия на хост открывается
      только после слова (приватность, батарея, работа при плохой сети).
- [ ] **Foreground service** (Android 14+, `foregroundServiceType=microphone|connectedDevice`) — тихий фон,
      систему не убивает.
- [ ] **Bluetooth-аудио:** захват мик гарнитуры (HFP/**SCO**) + вывод (**A2DP**); аккуратное переключение
      SCO↔A2DP. Запасная **кнопка на гарнитуре** вместо wake word.
- [ ] **Клиент сессии:** стрим аудио на Gateway, проигрывание TTS, barge-in, индикация (говорит/слушает/думает).
- [ ] **Выбор персоны** (Jarvis/Joi) и базовое управление — из клиента.
- [ ] **Батарея:** тайм-ауты сессии, лёгкий детектор; замер расхода.

**Проверка:** Криник в наушниках, телефон в кармане: «Jarvis…» → разговор; «Joi…» → мягкая персона; фон не
жрёт батарею катастрофически; перебивание работает.

## Фаза 5 — Мобильный прод: WebRTC + шлифовка «как с живым» 🔲

- [ ] **WebRTC** для аудио-канала (сабсекундная задержка, jitter buffer, эхо/шумоподавление, NAT traversal)
      — устойчивость «на ходу по городу». Кандидат инфраструктуры — **LiveKit** / Pipecat WebRTC. WebSocket
      остаётся для control-событий.
- [ ] Настоящие голоса: **Вихров** (Jarvis) и **девичий/бархатный** (Joi) — обучение на локальной GPU
      (см. idea 05, раздел голоса/личности).
- [ ] Шлифовка задержки и естественности: пороги VAD, длина первого чанка TTS, интонации.
- [ ] Интеграция **PAI** ([idea](../ideas/psycho_acoustic_interface/)): навык `/pai` доступен голосом через
      Gateway (запуск протоколов в наушники).

**Проверка:** прогулка по городу — разговор без ощутимых лагов и обрывов, интонации приятные, персоны
переключаются, PAI-протоколы запускаются голосом.

## Зависимости, риски, что переиспользуем

- **Уже есть в KLAS (переиспользовать, не переделывать):** Tailscale (транспорт), llama-server `/v1` (LLM),
  практика Bearer-ключей, homepage-пульт (тестовый веб-клиент и управление).
- **Заимствуем:** Whisper.cpp/faster-whisper (STT), Piper (TTS), Pipecat/LiveKit (realtime-оркестрация/
  WebRTC), HA Companion App / wyoming-satellite (референс Android-клиента), Wyoming/OpenAI-Realtime (форма API).
- **Риски:**
  - *Мобильные ограничения Android* (фон, батарея, BT SCO) — главная практическая сложность → Фаза 4
    отдельно, возможно форк готового клиента.
  - *Задержка сквозного каскада* — митигировать стримингом+barge-in с Фазы 1; при нужде — S2S-модель в будущем.
  - *Соблазн «обширного API»* — держать шлюз тонким; форма — стандартная; наружу только tailnet.
  - *Стек Node vs Python* — допускается гибрид (заметка владельца): realtime-аудио на Python, control на Node.
- **Главная зависимость:** базовый Jarvis (навыки, память, персоны Jarvis/Joi — idea 05). Фазы 1–2 можно
  начать раньше как фундамент, Фазы 4–5 — после базового Jarvis.
