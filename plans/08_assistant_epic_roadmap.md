## Сквозные принципы эпика

1. **Ядро — сначала** (Q1). Jarvis/PAI/Android не должны отвлекать от стабильного стека, основной модели
   и пульта. Но каждое решение по ядру принимаем, **помня о Jarvis** (не закрывать себе двери).
2. **Стек — по критериям, не по привычке** (⚠️-примечание): длинная поддержка > активность сообщества >
   скорость/экономичность на железе > простота/надёжность. Node — там, где естественен (control-API,
   интеграция с llama-server, homepage); Python — там, где зрелее (realtime-аудио, STT/TTS, ML).
3. **Тонкая стандартная граница, не «обширный API»** (research 07): Jarvis Gateway = realtime-голос +
   маленький control-API, форма по образу OpenAI Realtime / Wyoming.
4. **Приватность — стержень** (Q5): голос/микрофон/управление ПК — только в tailnet; данные не покидают хост.
5. **Заимствуем лучшее** (Q4 + заметки владельца): HA Companion App, Pipecat/LiveKit, Whisper.cpp, Piper,
   Wyoming — как база/референс, а не переписываем.
6. **Честность PAI**: ассистивный аудио-UX, не «программирование мозга», не медицина; безопасность —
   обязательный слой.

## Закладки в ЯДРО уже сейчас (то самое «чуть-чуть B» из Q1)

Пока развиваем ядро — оставляем «крючки», чтобы Jarvis потом встал легко и без переделок:

- [ ] **Homepage: место под web-тул управления ИИ-ассистентом.** При работе над пультом (Фаза 5 ядра)
      предусмотреть отдельный раздел/страницу «Ассистент» — **пока только для Криника, под паролем**
      (близким доступа нет; отделено от гостевого Open WebUI). Достаточно заглушки-плейсхолдера с
      правильным auth-контуром. → [plan 07 Фаза 2](07_jarvis_gateway_and_client.md).
- [ ] **llama-server OpenAI `/v1` — стабильный контракт.** Jarvis переиспользует его как «мозг» —
      не ломать совместимость эндпоинта при сменах модели/движка ядра.
- [ ] **Tailnet готов к устройствам.** Убедиться, что схема Tailscale допускает добавление носимого
      устройства Криника как узла (без публичного выставления голоса).
- [ ] **Конвенции памяти/навыков.** Память Jarvis (idea 05) и навыки-CRUD — держать в уме при выборе
      формата хранения/БД ядра, чтобы потом переиспользовать, а не изобретать заново.
- [ ] **Основная модель с адекватным контекстом и русским** (уже в работе, интервью 002) — фундамент
      «ума» Jarvis; голосовой ассистент наследует выбранную модель.

## Секвенс эпика (когда за что браться)

> Активируется **после** закрепления ядра (стабильный стек + основная модель + пульт). Порядок —
> от независимого-и-осязаемого к зависимому-и-сложному.

**Блок 1 — PAI-песочница** (автономно, не требует Jarvis; можно начать раньше остального эпика)
→ [pai_plan.md](../ideas/psycho_acoustic_interface/pai_plan.md) Фазы 1–2 + тренажёр (Фаза 3 PAI).
Генератор состояний + библиотека протоколов + слой безопасности. Даёт быстрый осязаемый результат.

**Блок 2 — Скелет разговора Jarvis на хосте** → [plan 07](07_jarvis_gateway_and_client.md) Фаза 1.
Каскад STT→LLM→TTS со стримингом/barge-in, локальный клиент. Доказывает «мозг+голос».

**Блок 3 — Jarvis Gateway + персоны + web-тул в homepage** → plan 07 Фаза 2. Тонкий API (realtime +
control), слой персон (Q2), тестовый веб-клиент = тот самый homepage-тул управления ассистентом.

**Блок 4 — Транспорт tailnet** → plan 07 Фаза 3. Устройство Криника как узел tailnet, Gateway только там.

**Блок 5 — Носимый Android-клиент** → plan 07 Фаза 4. Форк готовой базы (Q4), wake word/фон/Bluetooth,
персоны Jarvis/Joi.

**Блок 6 — Мобильный прод + интеграция PAI** → plan 07 Фаза 5. WebRTC, настоящие голоса (Вихров/Joi),
навык `/pai` голосом, шлифовка «как с живым».

**Блок 7 — PAI-тренажёр в связке** → калибровка откликов мозга под конкретного пользователя, коды PAI
адаптируются (см. pai_plan). Может идти параллельно Блокам 5–6, когда есть носимый вывод.

```
Ядро (сейчас) ──► [Б1 PAI-песочница] ──┐
                                        ├─► [Б6 мобильный прод + PAI] ──► [Б7 PAI-тренажёр]
   [Б2 скелет]─►[Б3 Gateway+персоны+пульт]─►[Б4 tailnet]─►[Б5 Android]─┘
```
