# /kaif-update — уважительное обновление-миграция из origin

В апстриме существует более новая версия KAIF (см. `/kaif-version`). Этот скилл поднимает фреймворк
проекта до неё **уважительно**: никогда не ломает проект пользователя, сохраняет его локальные
кастомизации и каждый контент-артефакт (`bugs/`, `interviews/`, `ideas/`, `homeworks/`).

> ⚠️ Это меняет обёртку фреймворка. Подтверди с человеком перед применением. Сначала закоммить, чтобы
> обновление было чистым, откатываемым диффом.

## Процедура

1. **Предполётная.** Прочитай `.kaif/kaif.json` (текущая версия, `origin`, `tracking`). Если
   `tracking` = `fork` — подтверди, что человек действительно хочет тянуть с официального origin
   (обычно обновляются со своего форка). Убедись, что рабочее дерево чистое (коммит/stash сначала).

2. **Забери целевую версию** из `origin`: новый `KAIF.md` (и/или релизный ассет). Самораспаковывающийся
   `KAIF.md` — единственный источник истины о том, что содержит новая версия.

3. **Дифф старое → новое.** Сравни руководящие документы, скиллы и конвенции нового фреймворка с
   развёрнутым. Классифицируй каждое изменение: (а) чистый апгрейд фреймворка (безопасно применять),
   (б) касается файла, кастомизированного пользователем (нужен аккуратный 3-way merge), (в) новая
   возможность (добавить).

4. **Мигрируй уважительно:**
   - Применяй апгрейды, принадлежащие фреймворку (скиллы, шаблоны руководящих документов) — перевыведи
     развёрнутую обёртку из нового `KAIF.md`, на рабочем языке и в сфере проекта (см.
     `.kaif/kaif.json`).
   - **Сохраняй кастомизации пользователя** — где он редактировал руководящий документ, мержи, а не
     перезаписывай; конфликты показывай человеку, не гадай.
   - **Никогда не трогай контент-артефакты** (`bugs/`, `interviews/`, `ideas/`, `homeworks/`) и
     собственные файлы проекта пользователя. Мигрируется только обёртка/ядро KAIF.
   - Освежи npm-handles `kaif:*`, если новая версия их изменила.

5. **Проставь новую версию** в `.kaif/kaif.json` (version, released). Прогони `npm run kaif:check`
   (валидатор), чтобы убедиться, что результат корректен.

6. **Отчитайся и закоммить.** Суммируй, что изменилось, что смержено, какие конфликты остались
   человеку. Коммит `chore: update KAIF to vX.Y (DATE)`.

## Заметки
- Ключевое слово — **уважительно**: проект должен оставаться целым и рабочим на каждом шаге.
- Если миграция большая или рискованная — делай её за чистым коммитом, чтобы легко откатить.
- Сильно разошедшемуся проекту может больше подойти форк (`/kaif-fork`), чем трекинг origin.
