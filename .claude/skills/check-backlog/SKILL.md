# /check-backlog — ревизия беклога (bugs/ и plans/) и DONE-теги

Со временем `bugs/` и `plans/` накапливают файлы, которые сделаны, но не помечены `DONE`, — и список
файлов больше не говорит, что осталось. Этот скилл наводит порядок: собирает актуальный открытый
список И помечает действительно закрытые файлы DONE.

Опирается на конвенцию DONE-тега-в-имени (см. `AGENT_GUIDE.md` → «Беклог и DONE-тег»).

## Что делать

1. **Собери все файлы беклога:**
   - `ls bugs/` — все документы багов.
   - `ls plans/` и `ls ideas/` — идеи/фичи и worklog'и/планы.
   - ⚠️ В корне и в `plans/` есть СПРАВОЧНЫЕ документы (не задачи): `MASTER_PLAN.md`, карты, `GOAL.md`
     и т.п. — их DONE НЕ помечать; это живые справочники, не закрываемые задачи. Помечай только
     конкретные баги/идеи/фичи/задачи.

2. **Раздели на два списка по имени файла:**
   - **Открытые (без `DONE` в имени)** — кандидаты в беклог «реализовать». Отметь их.
   - **Помеченные `DONE`** — уже закрыты, пропусти.

3. **Для КАЖДОГО открытого файла — определи реальный статус** (не гадай!):
   - Прочитай документ: есть ли внутри статус ✅/CLOSED/РЕАЛИЗОВАНО/ПОЧИНЕНО/ДОКАЗАНО?
   - Сверь со `STATUS.md` и, при надобности, с кодом/`git log` — действительно ли сделано?
   - **Если реально СДЕЛАНО и подтверждено** → выполни DONE-тегирование (шаг 4).
   - **Если НЕ сделано / частично / только исследование (🔧/🟡/🔬)** → оставь открытым, добавь в
     актуальный беклог (с короткой пометкой «что осталось»).
   - **Если неясно** → НЕ помечай; оставь открытым с пометкой «нужна проверка».

4. **DONE-тегирование (только подтверждённо закрытых):**
   - Переименуй, вставив `DONE` после номера, с сохранением истории:
     `git mv bugs/13_detach_crash.md bugs/13_DONE_detach_crash.md` (номер не менять; формат
     `<NN>_DONE_<name>.md`).
   - **Добавь статус-секцию внутрь документа**, например:
     ```
     ## ✅ STATUS: DONE (<дата>)
     Что сделано: <краткое резюме фикса/реализации>.
     Как проверено: <запуск/харнесс/измерение/хеш коммита/итерация цикла>.
     ```
   - Коммит (в цикле — по обычной дисциплине; иначе пакетом в конце ревизии).

5. **Выдай итог:**
   - Короткий список ОТКРЫТЫХ задач (беклог) с приоритетом: закончить начатое > баги > новые идеи.
   - Список файлов, помеченных `DONE` в этой ревизии.
   - Всё, что осталось как «нужна проверка» (неясный статус).

## Заметки
- Аккуратная ревизия: помечай DONE только РЕАЛЬНО закрытое и проверенное — лучше оставить открытым,
  чем ошибочно закрыть.
- Справочные документы никогда не DONE-тегируются.
- В автоциклах и при `/refresh-context` вызывай раз в несколько итераций, не каждую.
- Всплыли вопросы человеческого уровня — в `interviews/` + пометка `STATUS.md`, не решай вслепую.
